

1. Executive Summary
GraphHire is a next-generation job portal that leverages Neo4j's property graph database to deliver relationship-aware hiring experiences that are impossible with traditional relational databases. Instead of matching candidates to jobs through simple keyword filters, GraphHire understands the web of relationships between skills, experience, companies, roles, endorsements, and career trajectories — enabling smarter matches, better recommendations, and rich analytics for both candidates and employers.

This PRD covers the full feature set, data model, Neo4j implementation strategy, API design, and vibe-coding guidelines for building GraphHire from scratch.

2. Problem Statement & Opportunity
2.1 Current Pain Points
Keyword-based matching misses highly qualified candidates who use different terminology
No understanding of skill adjacency — a React developer is also likely proficient in JavaScript/TypeScript
Career trajectory and growth potential are ignored — all experience is treated as equal
Referral networks are invisible — existing portals do not map who referred whom or team overlaps
Duplicate data — the same company, skill, or university is stored inconsistently across millions of rows

2.2 Why Neo4j Solves This
A graph database models the real-world relationships in hiring natively. Every candidate, skill, job, company, and endorsement becomes a node; connections between them become first-class relationships with properties. This unlocks:
Multi-hop traversal: Find candidates who know someone at Company X, who have Skill Y, within 2 degrees
Pattern matching: Identify career paths that consistently lead to senior engineering roles
Real-time recommendation graphs: Cypher queries replace complex JOIN chains
Fraud detection: Spot fake endorsements via circular reference detection

6.4 Graph Data Science — Recommendations
Use the GDS library for advanced recommendation algorithms that go beyond simple Cypher:

Node2Vec Embeddings: Generate vector embeddings of Candidate and Job nodes based on graph structure. Use cosine similarity for 'candidates like this candidate' recommendations.
PageRank on Endorsements: Rank candidates by the influence of their endorsers — an endorsement from a Principal Engineer carries more weight than one from a recent grad.
Community Detection (Louvain): Identify natural talent clusters (e.g., 'ML Engineers who also do MLOps') to power skill-cluster browsing.
Shortest Path for Referrals: Use Dijkstra / BFS to find the fastest referral path between a candidate and a company.

6.5 API Design Principles
Backend services communicate with Neo4j via the official Neo4j Driver (JavaScript / Python). All queries are parameterised to prevent injection.

REST + GraphQL: Expose a GraphQL layer on top of Neo4j for flexible querying from the frontend
Cypher as Service: Wrap each query type in a named service function (e.g., getMatchingCandidates, getSkillGap)
Caching: Cache top-match results in Redis for 5 minutes; invalidate on profile/job updates
Pagination: Use Neo4j SKIP/LIMIT with keyset pagination on postedAt for infinite scroll
Rate Limiting: Graph traversal queries are expensive — rate-limit unauthenticated calls to 20/min

8. Security & Compliance
8.1 Data Security
All PII (email, phone, location) stored encrypted at rest in Neo4j using property-level encryption
Candidate profiles are private by default — visibility controlled by isPublic and visibleToCompanies flags on Candidate node
All Cypher queries use parameterised inputs — no string interpolation
Role-based access control: Candidate, Employer, Admin roles with different graph read/write permissions

8.2 Privacy & GDPR
Right to erasure: DETACH DELETE candidate node removes all relationships atomically
Data portability: Export candidate graph as JSON using APOC export procedures
Consent management: consentGiven and consentDate properties on Candidate node
Anonymisation: Analytics queries strip PII using graph projections

10.3 Key Design Decisions
Skills are shared nodes to prevent 'React', 'ReactJS', 'React.js' fragmentation. Use a seeded skill taxonomy on launch.: Skill normalisation
GraphHire is graph-native. No PostgreSQL mirror. All reads and writes go through Neo4j.: No SQL fallback
Nodes are never hard-deleted in production; use a deletedAt property and filter in queries. This preserves referral/history integrity.: Soft deletes
Stored on Assessment nodes (not on HAS_SKILL relationships) to allow re-testing without destroying history.: Assessment scores
